## 📂 מבנה תיקיות הפרויקט ב-Git (Project Structure)

הפרויקט בנוי בצורה מודולרית ונקייה בהתאם להנחיות האקדמיות של כל אחד משלבי הגשת הפרויקט:

*   **`DBProject/שלב א`:** תכנון בסיס הנתונים בתצורת 3NF, קובץ ERD, יצירת 10 טבלאות מחוברות בקישורים זרים (`createTables.sql`), ריקון מבנים (`dropTables.sql`), והזנת נתוני בדיקה מעל ל-1,000 שורות (`insertTables.sql`).
*   **`DBProject/שלב ב`:** שאילתות מורכבות ובו-זמניות (`Queries.sql`), הגדרת אילוצי תקינות מורכבים (`Constraints.sql`), יצירת אינדקסים לשיפור ביצועים (`Index.sql`), ניהול עסקאות (`RollbackCommit.sql`), וגיבוי בסיס הנתונים (`backup2.sql`).
*   **`DBProject/שלב ג`:** פיתוח מבטים מורכבים המשלבים קשרים לביצוע אופטימיזציה שיווקית ולוגיסטית (`Views.sql`), הרצת שאילתות מעל מבטים ללא מזהי ID ואינטגרציה לניהול שרשרת אספקה (`Integrate.sql`, `backup3.sql`).
*   **`DBProject/שלב ד`:** כתיבת קוד צד-שרת מורכב ב-PL/pgSQL: פרוצדורות חידוש מלאי ואופטימיזציית תקציב, פונקציות חישוב ROI שיווקי ודוחות מורכבים מבוססי `Ref Cursor` להזמנות ספקים, וטריגרים לאבטחה ורישום מלאי אוטומטי (`Log_Stock_Movement.sql`, `Enforce_Campaign_Budget_Policy.sql` וכו').
*   **`DBProject/שלב ה` (זהו דוח ההגשה הראשי!):** ממשק המשתמש הגרפי, מערכת הבקרה הכוללת, והוראות הפעלה.

## 🛠️ שלב ה' - הממשק הגרפי ומערכת הבקרה (GUI Panel)

בשלב זה בנינו ממשק משתמש (GUI) אינטראקטיבי יוצא דופן המאפשר שליטה חיה בנתוני הפרויקט ומדגים את כלל הדרישות האקדמיות תחת קורת גג אחת בעיצוב משוכלל וארכיטקטורה יציבה.

### 💻 דרך העבודה והכלים הטכנולוגיים בהם השתמשנו

כדי להבטיח ביצועים מהירים בקצב ענן וממשק אטרקטיבי, בחרנו להשתמש בכלים מתקדמים:
1.  **React 19 & TypeScript (strict mode):** בניית רכיבים מודולריים המאפשרים הגנה מלאה על סוגי נתונים (Type Safety), ניהול מצבי מסד נתונים בזמן אמת, ואינטגרציה מהירה.
2.  **Tailwind CSS (Vite Preprocessed):** עיצוב מותאם אישי ויוקרתי (Modern Premium Minimalist Slate) תוך שימוש במרווחים מדויקים, צבעי פירמה נקיים, וטיפוגרפיה מסוגרת (Inter ו-JetBrains Mono עבור פלטי הקוד).
3.  **Framer Motion:** החלת אנימציות מעבר חלקות ומיקרו-אינטראקציות חיות המעניקות תחושת זמינות וזרימה של אפליקציית קצה אמיתית.
4.  **Lucide React:** אייקונים וקטוריים מתקדמים לצורך שיפור הקריאות הוויזואלית.
5.  **Stateful PostgreSQL DB Engine Emulator:**  
    בממשק חקרנו ודימינו מנוע בסיס נתונים מצוין המבצע בזמן אמת את פעולות השרת כפי שנכתבו בשלבים א'-ד'. הוא מציג קישורי Joins אוטומטיים, יחסי גומלין ומחיקות משורשרות (למשל מחיקת מוצר מוחקת אוטומטית את הווריאציות שלו - Cascade Option), יצירת רשומות, ועדכון נתונים מבוסס מפתח מאוכלס מראש.

המערכת פועלת בשתי דרכים - הרצה ישירה בענן (מומלץ) והתקנה מקומית:

העלינו את האפליקציה לשרת ענן מאובטח ב-Cloud Run. המערכת פעילה לחלוטין ופתוחה לבדיקה מיידית בדפדפן:

*(הערה: הממשק מותאם לחלוטין לכלל הדפדפנים המודרניים וכולל רספונסיביות מלאה למחשבים וטאבלטים).*

---

### 💻 אפשרות ב': התקנה והרצה מקומית על המחשב (Local Setup)

אם ברצונכם להריץ את הממשק בסביבת הפיתוח המקומית שלכם, בצעו את השלבים הבאים בדיוק:

1.  **קבלת קוד המקור:** החזיקו בתיקיית הפרויקט המקומית שלכם.
2.  **התקנת חבילת התלויות (Node Package Manager):**  
    פתחו את חלון הטרמינל/PowerShell בתיקיית השורש של הפרויקט והריצו:
    ```bash
    npm install
    ```
    *(עבור משתמשי Windows הנתקלים בשגיאת Execution Policies בהפעלת npm, הריצו תחילה את הפקודה המאשרת `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser`)*.

3.  **הרצת שרת פיתוח מקומי:**  
    לאחר השלמת ההתקנה, הריצו את פקודת ההפעלה של דפדפן הפיתוח:
    ```bash
    npm run dev
    ```
4.  **צפייה מקומית:**  
    פתחו את הדפדפן שלכם בכתובת המקומית הבאה:
    🔗 [http://localhost:3000](http://localhost:3000)

### 4. מסך הרצת פרוצדורות ופונקציות PL/pgSQL של שלב ד' עם פלט Notice שרת
תיבת הרצה מרהיבה מדמה קונסולת קוד של שרת בסיס נתונים (SQL Terminal Logs). המשתמש מזין פרמטרים ומריץ את התוכניות שכתבנו בשלב ד':
*   **פרוצדורת רענון מלאי מחסנים (`sp_process_warehouse_restock`):** מחשבת חוסרים ומנפיקה הזמנה אוטומטית. הפעלתה מעדכנת את מלאי הווריאציות, מייצרת רשומות ומדפיסה הודעות שרת דינמיות.
*   **אופטימיזציית תקציב קמפיינים שיווקיים (`sp_adjust_campaign_budgets`):** מעדכנת תקציבים ואוכפת באמצעות טריגר חסימת תקציבים קריטיים במראה של חריגות מעל 50%.
*   **פונקציית דירוג ROI של קמפיין שיווקי (`fn_calculate_campaign_efficiency`):** מחשבת ואומדת את אחוז היעילות על פי ההנחות המשוקללות ברקע.
*   **דוח רכש מפורט מבוסס Ref Cursor (`fn_get_supplier_order_report`):** שולח מצביע מרובה שורות ומביא דוח נתונים עשיר מהטבלאות המשורשרות.

## 🛡️ אכיפת אילוצים וטריגרים הרצים ברקע במערכת

על מנת לענות על דרישות הבקרה התקינות בצורה אופטימלית, המערכת מיישמת ומריצה שני טריגרים עסקיים חיוניים:

1.  **Trg_log_stock_movement (טריגר מלאי):**  
    מופעל בעת כניסת מלאי חדש או שינוי מלאי של וריאציית בגד. הטריגר רושם לוג בטבלת הלוגים של המערכת, מתעד את הכמות הישנה לעומת הכמות החדשה, ומייצר התרעת "מלאי נמוך קריטי" עבור פריטים המכילים פחות מ-10 יחידות זמינות.
2.  **Trg_enforce_campaign_budget_policy (טריגר תקציבים):**  
    מופעל בעת עדכון או הזנה של תקציב קמפיין שיווקי חדש. הטריגר מוודא כי התקציב אינו עובר את תקרת ה-50% מהתקציב השנתי המוקצה למנהל שלו. במידת חריגה, המערכת פוסלת את החריגה הפיננסית, נועלת את התקציב על התקרה המקסימלית המותרת ברמה הלוגית ומתעדת לוג חריגה מפורט בטבלה הייעודית.
